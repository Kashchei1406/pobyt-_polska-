# Выгрузка сайта на сервер и проверка безопасности

## Состав проекта

1. Статический фронтенд:
   - `index.html`
   - `styles.css`
   - `main.js`
   - папка `assets/`
2. Backend-логика (Google Apps Script):
   - `backend/FormToSheetAndTelegram.gs` — принимает заявки с сайта, записывает в Google Таблицу и отправляет в Telegram
   - `backend/ReviewsFromGoogleMaps.gs` — отдельный endpoint для отзывов с Google Maps (отдельно разворачивается)

Сайт ожидает, что в `index.html` будут заданы:
- `window.FORM_SUBMIT_URL` (URL развёртывания `FormToSheetAndTelegram.gs`)
- `window.REVIEWS_API_URL` (URL развёртывания `ReviewsFromGoogleMaps.gs`, опционально)

## Как правильно выгрузить (статический хостинг)

### 1. Разверните Apps Script и получите URL

#### Заявки: таблица + Telegram
1. Разверните `backend/FormToSheetAndTelegram.gs` как **Веб-приложение**.
2. Получите URL вида `https://script.google.com/macros/s/.../exec`.
3. В `index.html` подставьте в `window.FORM_SUBMIT_URL` этот URL.

Проверка перед публикацией:
- откройте ссылку скрипта с `?test=1` (см. `docs/КАК_ПРОВЕРИТЬ_СКРИПТ_В_ТАБЛИЦЕ.md`)
- отправьте тестовую заявку с локального `http://localhost:5173`

#### Отзывы (опционально)
1. Разверните `backend/ReviewsFromGoogleMaps.gs` как отдельный проект Apps Script.
2. Включите Script Properties: `PLACES_API_KEY` и `PLACE_ID` (см. `docs/ОТЗЫВЫ_GOOGLE_MAPS.md`).
3. Подставьте полученный URL в `window.REVIEWS_API_URL` в `index.html`.

### 2. Подготовьте сервер для статики

Так как все ссылки в `index.html` относительные (`./styles.css`, `./main.js`, `./assets/...`), сайт должен быть опубликован в **корень домена** (или в место, где эти относительные пути не ломаются).

Что разместить на сервере:
- загрузить целиком папку проекта на веб-сервер
- убедиться, что доступны:
  - `GET /index.html`
  - `GET /styles.css`
  - `GET /main.js`
  - `GET /assets/...`

Важно:
- включите HTTPS (браузер может блокировать часть поведения при смешанном контенте)
- статические файлы должны отдаваться с корректными Content-Type (HTML/CSS/JS/SVG)

### 3. Пример (Nginx) для статики

Если вы размещаете сайт в директории, например `/var/www/pobyt/`, то базовая конфигурация может быть такой:

```nginx
server {
  listen 443 ssl;
  server_name example.com;

  root /var/www/pobyt;

  location / {
    try_files $uri $uri/ =404;
  }

  location ~* \.(css|js|png|jpg|jpeg|gif|svg|webp|woff2?)$ {
    expires 30d;
    add_header Cache-Control "public";
    try_files $uri =404;
  }
}
```

Если вы хостите без Nginx (Apache/Сервер-панель), логика та же: отдавайте `index.html`/статические файлы как есть.

## Как проверить после выгрузки

### 1. Формы
1. Откройте сайт на реальном домене.
2. Отправьте тестовую заявку (любой из 3 форм: hero / контакты / модалка из карточек).
3. Убедитесь, что:
   - в Google Таблице появилась строка на листе `Лиды`
   - в Telegram пришло сообщение (если настроено `BOT_TOKEN` и `CHAT_ID`)

### 2. Отзывы
1. Прокрутите до секции «Отзывы».
2. Если отзывы не грузятся:
   - проверьте `window.REVIEWS_API_URL`
   - откройте URL развёртывания `ReviewsFromGoogleMaps.gs` в браузере (должен отдавать JSON/JSONP)

### 3. Консоль и Network
Откройте DevTools → Console и Network:
- проверьте, что запросы к Apps Script возвращают 200
- проверьте отсутствие 403/404 для статических файлов `main.js` и `styles.css`

## Типовые ошибки при публикации

1. `Форма не отправляет заявки`
   - `window.FORM_SUBMIT_URL` не задан или старый
   - неправильно опубликован Apps Script deployment (нужен именно URL web app)
2. `Не применяется CSS/JS`
   - сайт опубликован не в корень, а относительные пути (`./...`) сломались
   - на сервере неверные permissions/права на чтение `assets/` или `main.js`
3. `Отзывы не подгружаются`
   - `window.REVIEWS_API_URL` пустой/невалидный
   - в Script Properties не заданы `PLACES_API_KEY` и `PLACE_ID`

## Ошибки и уязвимости (по текущей реализации)

Ниже — реальные риски, которые прослеживаются по коду. Они относятся к Security/Robustness, а не к “косметике”.

### 1. Telegram: небезопасный `parse_mode: 'HTML'` + пользовательский ввод

Файл: `backend/FormToSheetAndTelegram.gs`, функция `sendTelegram()`.

Там задаётся:
- `parse_mode: 'HTML'`
- в `text` добавляются значения `name`, `phone`, `city`, `message`, `source` напрямую из запроса пользователя

Риск:
- злоумышленник может отправить “HTML-теги”, и Telegram интерпретирует их как форматирование (инъекция разметки)

Что нужно сделать:
- перед сборкой `text` экранировать символы HTML: `& < > " '`
- либо отказаться от `parse_mode: 'HTML'` и использовать plain text

Минимальный принцип:
- любые поля, пришедшие от пользователя, нельзя вставлять в HTML-режиме без экранирования

### 2. Google Sheets: formula injection (CSV/формулы)

Файл: `backend/FormToSheetAndTelegram.gs`, где делается `sheet.appendRow(row)`.

Риск:
- если пользовательское поле начинается с `=`, `+`, `-`, `@`, Google Sheets может трактовать это как формулу
- при последующей обработке/экспорте это может привести к неприятным эффектам

Что нужно сделать:
- перед `appendRow` проверять каждую строку и при необходимости префиксить `'` (одиночная кавычка) для значений, которые могут быть формулами

### 3. Публичный endpoint без защиты от спама/подделок

Файл: `backend/FormToSheetAndTelegram.gs`, функции `doPost/doGet`.

Текущая модель:
- web app endpoint доступен публично
- нет:
  - токена/секретного ключа
  - rate limiting
  - captcha/hcaptcha
  - строгой валидации входных данных

Риски:
- спам запросами в Таблицу и Telegram
- “подмена отправителя” (CSRF по смыслу: злоумышленник может инициировать POST со своей страницы)

Что нужно сделать:
- добавить проверку секретного значения (например, `secret` в запросе, заранее известный фронту и backend)
- ограничить частоту по IP/ключу (если есть возможность)
- добавить captcha на клиенте (минимум hCaptcha/Turnstile)

### 4. `doGet` с `?test=1` пишет в таблицу

Файл: `backend/FormToSheetAndTelegram.gs`, `doGet(e)`.

Сейчас:
- любой, кто знает URL deployment, может вызвать `?test=1` и писать тестовые строки

Риск:
- загрязнение таблицы
- косвенно — спам/нагрузка

Что нужно сделать:
- убрать `?test=1` после проверки
- или защищать тест-режим (секрет, whitelist, admin-only)

### 5. Reviews: JSONP endpoint (скрипт-тег) и отсутствие ограничений по частоте

Файл: `backend/ReviewsFromGoogleMaps.gs`, `doGet` возвращает JSONP при наличии `callback`.

Что сейчас хорошо:
- callback валидируется регулярным выражением, что снижает риск произвольного JavaScript-инъекта в URL

Остаётся:
- endpoint может быть использован для дорогих вызовов Places API (лимиты/стоимость)
- нет rate limiting

Что нужно сделать:
- повесить простую защиту (секрет/ключ/минимальное ограничение)
- кеширование у вас уже есть — но стоит дополнительно ограничить частоту для одного источника

## Рекомендации “минимум чтобы стало безопаснее”

Если нужно быстро улучшить безопасность без капитального рефакторинга:

1. В `sendTelegram()` переставить `parse_mode` на plain text или сделать HTML-экранирование входных данных.
2. В `doPost()` добавить sanitize для значений перед `appendRow()` (защита от formula injection).
3. Добавить секретный параметр/токен, который проверяется на backend перед записью и отправкой в Telegram.
4. Удалить/защитить `?test=1`.
